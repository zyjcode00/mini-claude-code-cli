## 换行符处理规范
- **重要**：edit_file 工具已支持自动归一化换行符，可以正确处理 CRLF（Windows）和 LF（Unix）文件。
- **最佳实践**：在调用 edit_file 时，无需关心目标文件的换行符类型，工具会自动处理。

## 文件编辑工具选择规范
- **强制性**：根据文件长度和修改复杂度选择合适的工具，避免匹配失败。

### 工具选择决策树：
1. **文件长度 < 100 行**：
   - 优先使用 `write_full_file` 全量覆盖，简单直接，无匹配失败风险
   
2. **文件长度 > 100 行，且修改位置明确**：
   - 使用 `edit_file` 进行局部修改
   - **关键技巧**：使用 `read_file(raw_mode=True)` 获取原始内容，避免拷贝错误

3. **连续匹配失败 2 次以上**：
   - 立即切换到 `write_full_file`
   - 使用 `read_file(raw_mode=True)` 读取完整内容，手动修改后全量写入

### 最佳实践：
- **读取内容时**：优先使用 `raw_mode=True` 参数，输出无装饰符的原始内容，方便拷贝
- **编辑长代码块时**：old_str 应包含足够的上下文行（至少上下各 2 行），确保唯一性
- **遇到匹配失败时**：
  1. 使用 `read_file(raw_mode=True)` 检查原始内容
  2. 对比 old_str 和实际内容的差异
  3. 若无法快速定位问题，立即切换到全量写入模式

## 全局搜索规范
- **强制性**：在进行跨文件重构或修改核心接口前，必须先调用 `search_code` 进行全局影响评估，了解受影响的文件范围。
- **最佳实践**：使用 `search_code` 时，建议设置 `context=2` 参数以查看上下文行，避免误判。

## 符号表查询规范
- **强制性**：在修改任何核心逻辑前，必须先调用 `list_all_symbols` 或 `find_symbol_definition` 确认影响范围，避免盲目修改。
- **工具使用**：
  - `list_all_symbols`：快速查看项目所有 API 的大纲，适合理解陌生模块的整体结构。
  - `find_symbol_definition`：根据名称精准定位代码块（文件、行号、代码片段），跳过盲目的 grep 搜索。
- **最佳实践**：在重构前，先用 `find_symbol_definition` 查找目标符号及其所有引用，评估修改的影响面。

## 会话管理规范
- 当检测到任务跨度超过 30 轮对话时，主动建议用户使用 `manage_plan` 更新状态并开启新会话。
- 在完成重大重构任务后，必须调用 `write_file` 将核心逻辑总结到 `docs/arch_notes.md` 中，实现长效记忆。

## 测试驱动规范 (TDD)
- **强制性**：任何对 `core/` 或 `tools/` 的逻辑修改，必须伴随一个对应的测试脚本（如 `tests/test_xxx.py`）。
- **任务定义**：只有当 `run_pytest` 返回 "✅ 测试通过" 时，该任务才算真正完成。
- **自愈逻辑**：若测试失败，必须优先分析 Traceback，使用 `ReadTool` 读取受影响的代码，并通过 `FileEditTool` 修复，直到测试通过。

## Git 仓库保护规范
- **绝对禁止**：在任何情况下，都**禁止删除或破坏 `.git` 目录**。这是版本控制的核心基础设施。
- **状态优先**：当需要判断项目状态时，优先使用 `get_git_status` 工具，而不是手动检查文件。
- **自动保险**：
  - 每完成一个 Plan Step，系统会自动创建 Git 快照（commit）。
  - 当检测到 `edit_file` 在同一文件连续失败 2 次时，系统会自动回滚到上一个快照。
- **回滚安全**：使用 `git_rollback` 时，只会回滚代码内容，不会删除 `.git` 目录。
- **最佳实践**：
  - 在执行高风险操作前，建议先调用 `create_snapshot` 手动创建保险点。
  - 如果需要撤销修改，优先使用 Git 回滚机制，而不是手动删除文件。
